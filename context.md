# Context for Shema

This file exists to bring a new Claude chat or Cowork session up to speed on this project without re-deriving decisions that have already been made. Read this before making product, design, or architecture calls here — it captures the *why*, not just the *what* (the *what* is in `README.md` and the code itself).

## End goal

Ship Shema as a **paid app on the Apple App Store and Google Play Store**. Target user: pastors and teachers who prepare a sermon on one device (PC/Mac, or a tablet) and then present it on whatever device is in hand at the venue (phone, tablet, or the same laptop) — so cross-device continuity of the sermon itself is a core requirement, not a nice-to-have.

## Business model

- **Standalone one-time purchase**: local-only use, no account, no cross-device sync. This is the base product and must stay fully functional forever — sync is additive, never a requirement to use the app.
- **Monthly subscription add-on**: unlocks cross-device sync (the same sermon library available signed-in on any device). Gated per-account, not per-device.
- No Apple Developer account exists yet (as of this writing). Packaging/store submission work is intentionally deferred until the app itself is in a good place — see "Roadmap" below.

Recommended pricing (from competitive research, see "Competitive positioning" below): **one-time unlock ~$29.99–39.99**, **sync subscription ~$4.99/mo or ~$39.99/yr**. Not yet finalized — revisit once closer to launch.

## Design principles

- **Offline-first, always.** A pastor walking into a venue with no signal must have a fully working app. Sync (once built) reconciles in the background and never blocks Prepare or Present.
- **"Simple, graceful" visual theme.** Recent UI work has consistently traded density for calm — e.g. the Prepare-screen redesign moved the sermon title out of the main view into a popup (reopenable by tapping the title in the top bar) and collapsed the formatting toolbar into a toggleable sidebar, specifically because the prior layout felt "crowded" and off-theme. Default to this instinct: when adding a feature, ask whether it can live behind a toggle/popup rather than adding permanent on-screen chrome.
- **Present mode is for actually preaching from, not just displaying text.** Large type, a stage countdown timer, minimal chrome. This view's needs (can the presenter tell there's more content below the fold without looking away from the audience?) are treated as a distinct design problem from the editor's needs — decisions that are fine in Prepare (e.g. hiding a scrollbar for cleanliness) are evaluated separately for Present, where the same choice has a live-presentation-risk dimension.
- **Complementary to ProPresenter, not competing with it.** Shema exports to ProPresenter rather than trying to replace a church's existing projection/worship-presentation software. This is a deliberate go-to-market wedge ("the sermon-writing companion to the presentation software you already run"), not an accident of scope.
- **Single-file app, kept diffable.** `index.html` holds all markup/CSS/JS by design (no build step, trivial to serve/test). To keep it reviewable as it grows: icons live in `icons/` as standalone files (not inlined base64) with a `data-light`/`data-dark` attribute convention for theme variants, and fonts are self-hosted as base64 `@font-face` rules rather than pulled from a CDN, so the app has zero runtime network dependency for its own UI.

## Current architecture snapshot

- Plain PWA: `index.html` + `sw.js` (service worker, versioned `CACHE_NAME`) + `manifest.json`. No backend, no accounts.
- Data lives in `localStorage` only (see `README.md` for the exact keys). No server round-trip exists yet.
- As of the "Phase 1" work (see History), the data model is deliberately **sync-ready but not yet synced**: sermons and illustrations use UUIDv4 ids, carry `updatedAt`/`schemaVersion`, and deletes are tombstones (`deleted:true`) rather than array removal, specifically so a future sync engine can propagate a delete instead of resurrecting it. Backup export/import already does real last-write-wins merge by id+`updatedAt` as a preview of that logic on a single device.
- Responsive design has explicit breakpoints (480/768/1000/1600px) and, as of the most recent change, wide-screen content columns use **fixed pixel margins from the screen edge** rather than a centered `max-width` cap — content keeps growing on ultra-wide monitors instead of floating as a narrow centered island.
- Practice Mode (see History, 2026-07-20) introduced the app's first use of **IndexedDB** (`shema_practice` db) alongside `localStorage` -- audio recordings are real Blobs, too large and the wrong shape for `localStorage`'s string-only, ~5-10MB-ceiling model. Kept intentionally separate from every other `persist*`/`saveJSON` local-storage path and from the Supabase sync engine: practice audio never syncs, never leaves the device, by design.

## Competitive positioning (researched mid-2026, re-verify before acting on pricing)

The direct "sermon-outlining app" niche is thin. The closest real competitor is **Sermonary** (web-only, $49–99/mo, AI-writing-first, no confirmed ProPresenter integration) — priced as a premium outlier. **Preach: Sermon Builder** is the closest small native-app peer (free w/IAP, modest but real traction). Everything else is adjacent, not competing head-on: **Proclaim** and **ProPresenter** are whole-team worship-production software priced per seat ($18–72/mo, $29–59/mo); **Logos** bundles a sermon builder into its $9.99–19.99/mo study-suite subscription rather than selling it alone; **Planning Center** is team scheduling/ops, priced by headcount.

This is real whitespace, but cuts both ways: nobody has made a big hit out of "just sermon writing + present mode," which could mean it's unexecuted opportunity, or could mean pastors don't want to pay separately for a point solution when a bundled tool (Logos, Proclaim) already covers something similar. Treat this as an open risk, not a solved question — validate with real users before over-investing in features nobody asked to pay for separately.

Biggest go-to-market barriers identified (see full analysis from the market-evaluation session): near-zero organic App Store discoverability against incumbents with existing installed bases; pastors as a slow, trust-driven, often committee-gated buyer; no track record yet (cold start); growth partly tethered to ProPresenter's own visibility given the export-hook strategy; feature-race risk against a funded competitor (Sermonary) if differentiation drifts away from simplicity/offline-reliability/price.

## Roadmap

Phased plan to get from the current local-only PWA to synced, store-ready apps. Each phase is described in more depth in the conversation history that produced this file; summarized here for continuity.

1. **Data model sync-readiness — ✅ done.** UUIDv4 ids, tombstone deletes, per-record `schemaVersion`, persisted `deviceId`, last-write-wins backup merge. See History below for the commit.
2. **Accounts & entitlement — ✅ auth done, entitlement-writing intentionally deferred to Phase 4.** Backend decision made: Supabase. Google OAuth and email (magic link) sign-in are live, wired into the app, and confirmed with a real human click-through (not just a mocked test) — signed in successfully as a real Google account, landed back in the app already authenticated, entitlement display correctly read "Not yet" / "Not active". Sign in with Apple still deferred until an Apple Developer account exists (remember: Apple requires it the moment any other third-party login is offered, so it must land before App Store submission, not before then). `profiles` table (`owns_app`, `sync_subscription_active`, one row per `auth.users` id, auto-created via trigger on signup) is live with RLS locked to read-only for the client — entitlement flags are deliberately never client-writable, since an UPDATE policy on those columns would let a signed-in user grant themselves the paid unlock for free; they'll only ever be set by a server-side purchase-verification webhook (Phase 4, via the service_role key, which bypasses RLS) — that webhook, not this phase, is what will actually flip these flags to true. App side: `vendor/supabase.js` (self-hosted, precached, not CDN-loaded), a full Account management screen reachable from a new Settings screen (see below) with sign-in/out, an account-type badge, entitlement status, and the typical placeholder buttons a subscription screen needs (Manage Subscription, Restore Purchases, billing date/amount) ahead of Phase 4 actually wiring them up.

Also introduced a **Settings screen** (gear icon, replacing the old person-icon Account button in the brandbar) as the home for app-wide, infrequent actions -- Account, Backup, Restore from Backup, and Dark Mode all live there now, decluttering the Library header down to just "+ New". Import stays in Prepare, deliberately not moved here: it's a per-sermon authoring action tied to the editor, not an app-wide setting, and a `local` user needs it exactly as much as a `sync` user does.

**Account types** (client-computed, not a new schema column): `local` (no session -- the permanent default), `dev` (signed in as one specific hardcoded email -- a display label only, not a real permission boundary, since there's nothing sensitive gated behind it yet), `sync` (signed in with `sync_subscription_active` true -- nobody yet, since nobody can pay until Phase 4), `tester` (everyone else who's signed in -- which, practically, is *everyone* right now, since paying isn't possible yet; "tester" is simply what a signed-in, non-dev, non-paying account means during this pre-launch window).
3. **Sync engine — 🚧 in progress, sermons done, illustrations/templates next.** Decided the open conflict question from the original roadmap note: not silent LWW. The same record edited on two devices between syncs never silently drops an edit -- the remote version becomes canonical under the original id, and the local version is preserved as a new record (title suffixed "(synced copy)", `conflictOf` pointing at the original), surfaced with a small tag on its card. Reasoning: sync is invisible/automatic by design (that's the point of paying for it), so nobody is watching a merge happen the way a manual backup-restore user is -- a silent loss would go unnoticed, and a blocking "resolve this now" prompt could land at the worst possible moment (a pastor about to walk on stage). Deferring the decision to whenever the user happens to open the app, with no forced UI, fit better than either alternative. Increasing sync frequency was considered as a mitigation (shrinks the window where two online devices diverge) but doesn't touch the offline-divergence case at all, so it's a complement, not a fix -- not built as a separate lever yet (sync currently fires on every local save, debounced 2.5s, plus every 60s and on regaining connectivity, which is already fairly aggressive).
   - New Supabase tables, RLS same shape as `profiles` but full owner CRUD (not read-only) since this is the user's own content, not an entitlement flag: `sermons`, `illustrations` (with an `image_path` column pointing at a new private `illustration-images` Storage bucket rather than storing base64 in a column), `templates`.
   - Client-side sync engine (in `index.html`, right before INIT): per-record last-synced timestamps in a new `shema_sync_meta_v1` key, a generic `syncOneTable(cfg)` pull-then-push pass (currently wired for `sermons` only), triggered by `persistSermons()` (debounced 2.5s via `scheduleSync()`), a 60s interval, the `online` event, and auth state changes. Local storage stays authoritative always -- sync only ever mirrors it outward/inward, never blocks a local read/write, and runs for any signed-in user (not gated on `sync`-type entitlement, since nobody can pay yet).
   - Verified with a mocked-Supabase Playwright script covering all three cases: push a new local record, pull a remote edit, and the true-conflict case (edit the same record on both "sides" between syncs) -- confirmed the conflict path creates the tagged copy instead of dropping either edit.
   - Illustrations and templates now sync the same way. Illustration images ride as a Storage object rather than a jsonb/text column: `imagePath` is a lightweight local cache hint (which Storage path this device has already uploaded/downloaded) separate from `imageData` (the actual base64, always the offline-first source of truth, never cleared by a pull) so unrelated pushes don't re-upload unchanged image bytes. Fixed a real bug this surfaced: template deletion was array-removal (`templates.filter(...)`), not a tombstone like sermons/illustrations -- under sync that would have made a deleted template silently reappear (remote still has the row, this device just "hasn't seen it," so a pull would re-add it). Switched it to `deleted:true` + `renderTemplateList()` filtering it out, and updated "Restore built-in templates" to un-delete a tombstoned built-in in place instead of assuming the id is gone.
   - The `templates` table needs one more column the original migration missed: `desc` (templates carry a `name` *and* a `desc`, and the first migration only added `name`). Run `alter table public.templates add column if not exists "desc" text;` in the Supabase SQL editor before this reaches a device with existing templates to sync.
   - Verified with a second mocked-Supabase Playwright pass: illustration push uploads the image to the bucket and stamps `image_path` on the row, template push preserves `desc`, and pulling a remote-only illustration downloads and reconstitutes its image into local `imageData`.
   - Fixed a real, longstanding, sync-unrelated bug found while testing this live: inserting an illustration into a sermon (the picker) only ever copied its text onto the sermon item, never its image -- and no renderer (editor item list, Present's stage, manuscript/RTF export) had any code to display an image inline in sermon content at all, so a pictured illustration's image was silently dropped the moment it was used inside a sermon. Now the sermon item carries `imageData` too, with a small thumbnail in the editor's item row and a larger image above the text in Present. Manuscript/RTF export still don't show it -- lower priority, not requested, revisit if it comes up. Worth remembering: sermon `sections` sync as one jsonb blob per sermon, so an image used inside a sermon now rides along inside that column rather than through the illustration-images Storage bucket -- the same base64-bloat consideration flagged for illustrations applies here too and isn't solved yet.
   - Also surfaced a real gap during live two-device testing: illustrations pushed from one device weren't reliably reaching the other device's Illustrations library (separate from the sermon-item image issue above -- the illustration record itself, standalone in the library, didn't sync). Every sync error was previously swallowed silently by design (best-effort, never spam the user with sync noise) -- added `console.error` logging at each failure point (select, upsert, image upload, image download, and the outer per-pass catch) so a real failure is at least visible in devtools instead of invisible. Root cause not yet confirmed -- likely candidates: a device still running the pre-illustration-sync service worker version (very plausible, since that code landed one commit after sermon sync did and this app requires an actual reload to pick up a new SW version), or a real Storage/RLS issue on the illustration-images bucket specifically (the one path that isn't plain Postgres RLS, so it's the one thing the earlier mocked tests structurally couldn't validate). Needs a live retry with both devices confirmed on the latest deploy before concluding which.
   - Root cause found via the new logging: some local records (from before `crypto.randomUUID` was consistently available -- it needs a secure context, so early testing over plain HTTP or an older browser fell back to `uid()`'s non-crypto path) never actually had real UUIDs for ids, despite Phase 1's roadmap note claiming "UUIDv4 ids." The `sermons`/`illustrations`/`templates` tables were created with a strict `uuid` primary key, so Postgres rejected every push of one of these legacy-id records outright (`22P02 invalid input syntax for type uuid`) -- silently, once error logging was added it showed clearly. Fixed at the schema level, not the client: `id` (and the self-referencing `conflict_of` / `source_illust_id` columns) changed from `uuid` to `text` on all three tables, since the app already treats ids as opaque strings everywhere and never needed real UUID typing at the database layer. This was very likely also the illustration-not-syncing bug from the previous entry -- same failure mode, same fix.
   - Added the resolution step the earlier design note said would come "whenever the user gets to it": every conflict-copy card (sermon, illustration, template) now has two extra actions alongside the existing Edit/Delete -- "Keep This" (overwrite the original with this version's content and retire this copy) and "Keep Both" (clear the conflict flag, strip the "(synced copy)" title suffix, and let both stand permanently as separate records). The existing Delete button already covered the third case (discard this copy, leave the original untouched), so no new delete-specific logic was needed. "Keep This" only appears if the original record can still be found (not itself deleted); otherwise only "Keep Both" shows, since there's nothing left to replace. Templates track `conflictOf` locally for this same UI, but it isn't pushed to Postgres (the `templates` table has no `conflict_of` column) -- low priority to add given how rarely a template would actually be edited concurrently on two devices; can revisit if it turns out to matter.
   - Confirmed live by the user on real devices: the `uuid`→`text` id-column migration cleared the stuck sermons/illustration, and the Present-mode illustration-image fix both work correctly across two real devices. Phase 3's core sync engine is functionally done; remaining known gaps are the two lower-priority items noted above (sermon-embedded image storage-bloat, template `conflict_of` not in Postgres), not required before moving on.
   - Settings and Account, previously bottom sheets like every other modal in the app, are now a right-side drawer (full height, slides in over a dimmed backdrop, dismissible by tapping the backdrop) -- kept as an additive `.drawer-right` modifier class so no other modal's behavior changed. Moved the gear icon to the right edge of the brand bar to match (it's `margin-left:auto` now instead of just trailing the wordmark) since a right-aligned trigger reads more naturally as "opens a right-side panel" than a left-aligned one did. Also gave Version History and Import Notes the same drawer treatment (Import keeps a wider 640px cap, override via an inline style since it needs its toolbar+editor room) -- picked those two specifically because both are a "sit and work/read for a bit" experience like Settings/Account, unlike the quick-pick/quick-form modals (template picker, sermon info, illustration add/insert) that stayed as bottom sheets on purpose.
   - Closed out both bloat/visibility gaps flagged above. Sermon-embedded illustration images now follow the exact same Storage-object pattern as the illustrations table (`sermonBeforePush`/`sermonAfterPull`), keyed by the sermon item's own id under `${userId}/sermon-items/${itemId}.jpg` (not the source illustration's id, since the same illustration can be inserted into several items/sermons, each needing an independent copy that survives the original being edited or deleted) -- `sections` sent to Postgres now always has `imageData` stripped per item, `imagePath` is the only thing that rides in the jsonb column. And sync is no longer a black box: `lastSyncInfo` tracks the last pass's outcome, surfaced in the library header (replacing, for a signed-in user only, what used to be the manual-backup nag -- see next bullet) and re-rendered on every pass start/finish so "Syncing…" / "Synced Xm ago" / "Sync error · Xm ago" is always current. A "Sync Now" row appeared in Settings (visible only when signed in) wired to a `manualSyncNow()` that skips the normal 2.5s debounce but is still hard rate-limited to once per 5 seconds.
   - Decided the "Last backup" library-header question explicitly: a signed-in user (sync running) now sees sync status there instead of the manual-backup nag, since showing "Never backed up"/stale-backup warnings to someone who's actually fully protected by continuous cloud sync was actively misleading, not just redundant. The Settings screen's own "Restore from Backup" row is untouched and still always shows the manual-export status regardless of sync state -- a portable JSON copy is still worth having even when synced, so that meaning wasn't overloaded.
4. **Store packaging — not started.** Capacitor recommended for wrapping the PWA for both iOS and Android from one codebase (a Trusted Web Activity would only cover Android). RevenueCat recommended to unify StoreKit + Play Billing behind one API and offload receipt/subscription validation, rather than building that verification backend by hand.
5. **Store compliance — not started.** Privacy policy + ToS (required once accounts/data leave the device), Apple privacy nutrition label, Play Data safety form, App Store Review Guideline 3.1.1 (StoreKit-only for this app's subscription — no external payment links are allowed for this category).
6. **Migration & rollout — not started.** Existing local-only users need a one-time "sign in to turn on sync" flow that uploads their existing local sermons rather than starting them empty. The no-account local-only path must stay fully functional as the permanent default, not a crippled trial.

**Testing costs**: most of phases 2–4 can be built and fully tested for $0 before committing any money — Supabase/Firebase free tiers are indefinite (not trials), RevenueCat is free under $2,500/mo tracked revenue, Capacitor is free/open-source, Xcode's local StoreKit Configuration file simulates the entire purchase/subscription lifecycle with no paid Apple account, and Android has free sideload testing plus reserved test product IDs for billing flows. The two genuinely unavoidable costs are the Apple Developer Program ($99/year, needed for TestFlight/App Store listing) and Google Play Console ($25 one-time, needed for any Play Store distribution) — both can and should be deferred to the very end of the roadmap, once everything else already works.

## Working conventions

- Development happens on a feature branch (`claude/app-store-paid-launch-1jkhur` as of this writing); the repo owner reviews/merges to `main` themselves. Don't push to `main` directly.
- No build step, no test framework in the repo. Verification during development has relied on headless Playwright scripts (spun up ad hoc against a local `python3 -m http.server`) to check layout, drag-and-drop, and console errors across viewport sizes — there's no persisted test suite, so re-verify UI changes manually/via a quick script rather than assuming coverage exists.
- Before making a UI/UX call that trades off discoverability, affordance, or live-presentation risk (anything touching Present mode especially), reason explicitly about who's affected and how — this project's design decisions have consistently favored a short written risk/tradeoff evaluation over silently picking a default.
